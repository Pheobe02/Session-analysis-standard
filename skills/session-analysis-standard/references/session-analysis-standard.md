# 线上用户会话分析标准

## 1. 目标

每天基于线上会话导出文件，分析真实用户从开启会话到启动采集、采集成功、导出的完整链路表现，并识别未完成采集的关键问题会话，形成可汇报、可复盘、可追踪的问题分析结果。

每日固定输出两类内容：

1. 会话分析汇总报告
2. 问题会话明细表

## 2. 分析前置规则

所有统计前，先排除内部账号。

当前已知内部账号：

| 类型 | 值 |
| --- | --- |
| 用户Id | `43696d99-7de7-40b9-bf12-2b78a62635f7` |
| 邮箱 | `yupuqi@skieer.com` |

如果文件中包含用户名字段，也需要支持按内部用户名排除。后续新增内部账号时，统一加入内部账号名单。

汇总报告中必须展示：

- 原始会话数
- 排除内部账号会话数
- 纳入统计会话数

所有指标均基于“纳入统计会话数”计算。

## 3. 核心指标口径

| 指标 | 口径 |
| --- | --- |
| 开启会话数 | 排除内部账号后的全部会话数 |
| 启动采集成功会话数 | `启动次数 > 0` 的会话数 |
| 采集成功会话数 | `采集数据量 > 0` 的会话数 |
| 未启动采集会话数 | `启动次数 = 0` 的会话数 |
| 启动但未采集成功会话数 | `启动次数 > 0` 且 `采集数据量 = 0` 的会话数 |
| 导出成功会话数 | `导出次数 > 0` 或 `导出数据量 > 0` 的会话数 |

转化率指标：

| 指标 | 公式 |
| --- | --- |
| 启动采集会话转化率 | 启动采集成功会话数 / 开启会话数 |
| 启动采集到数据成功率 | 采集成功会话数 / 启动采集成功会话数 |
| 端到端采集成功率 | 采集成功会话数 / 开启会话数 |

端到端采集成功率也可以表达为：

```text
启动采集会话转化率 * 启动采集到数据成功率
```

## 4. 问题会话定义

问题会话包括：

- 未启动采集的采集意图会话
- 启动但未采集成功的会话
- 采集过程异常中断的会话
- 用户未继续确认导致未完成的会话
- 存在 Agent 理解、流程、网页限制、权限、额度、异常 bug 等问题的会话

同时保留两种口径：

| 口径 | 定义 |
| --- | --- |
| 广义问题会话 | 所有未完成端到端采集的会话，包括用户中断、未回应、非采集需求、积分问题等 |
| 有效产品 / Agent 问题会话 | 排除非采集需求、纯用户主动放弃、正常额度不足等情况后，真正需要产品、Agent、工程或体验优化处理的会话 |

## 5. 固定问题分类

默认按以下分类归因：

1. 用户主动中断 / 放弃
2. 生成预览数据后用户未回应
3. 采集数据失败
4. Agent 理解或流程问题
5. 非采集需求
6. 积分 / 权限 / 额度问题
7. 用户场景不支持
8. 目标网站登录 / 权限限制，需用户介入
9. 目标元素定位失败 / 页面结构识别失败
10. 异常 bug
11. 能力缺口 / 后续迭代需求
12. 正常完成 / 非问题会话

如果实际会话明显不属于以上分类，可以新增临时分类。新增分类需要在报告中说明新增原因，并判断是否建议纳入长期固定分类。

## 6. 关键归因原则

归因优先看根因，不只看最后一条报错。

如果最后报错是积分耗尽，但前面存在 Agent 重复探索、重复预览、流程不推进等异常行为，则归为“Agent 理解或流程问题”，而不是“积分 / 权限 / 额度问题”。

可以额外添加结果标签：

```text
异常积分耗尽 / 流程空转导致额度耗尽
```

“已生成预览数据”默认不等于异常。

| 情况 | 归类 |
| --- | --- |
| 生成预览数据后，用户没有继续回应 | 生成预览数据后用户未回应 |
| 用户多次 Continue / 明确要求继续，但 Agent 反复生成预览、不推进流程 | Agent 理解或流程问题 |
| 预览后用户输入 Start Extraction，并成功采到数据 | 正常完成 / 非问题会话 |

空回复规则：

| 情况 | 归类 |
| --- | --- |
| 最后一轮 Agent 回复为空，且系统含义是用户关闭插件 / 手动中断 | 用户主动中断 / 放弃 |
| 中断前存在页面加载失败、反复探索、预览异常、用户困惑、等待时间过长等信号 | 主分类仍按实际情况判断，并标注“是否需要人工排查 = 是” |

## 7. 分类判定补充

### 目标网站登录 / 权限限制，需用户介入

适用于 LinkedIn、Facebook、Google 登录页、OAuth、权限确认、登录墙等场景。核心特征是需要用户登录、授权或手动处理前置条件。

### 目标元素定位失败 / 页面结构识别失败

适用于页面可访问，但 Agent 找不到列表、按钮、翻页、目标字段，或反复找错元素的场景。

### Agent 理解或流程问题

适用于 Agent 没有理解用户目标、没有正确推进流程、反复预览、没有引导启动采集、或把用户意图转成了错误操作路径。

### 能力缺口 / 后续迭代需求

适用于用户需求合理，但当前产品或 Agent 暂不支持，不属于当天 bug，也不是用户误操作。

## 8. 问题分类汇总要求

每个问题分类在汇总报告中需要输出：

- 问题分类
- 数量
- 占全部纳入统计会话比例
- 占问题会话比例
- 典型 case
- 用户网址
- 简要原因

典型 case 至少包含：

- session id
- 用户场景
- 用户网址
- 问题摘要

## 9. 排序规则

在每日汇总报告中，以下部分必须按问题出现频次由高到低排序：

- 问题分类统计
- 各分类典型 case
- 有效产品 / Agent 问题清单
- 需要人工排查的会话分类汇总

排序口径：

1. 优先按该问题分类的会话数量降序排列。
2. 如果数量相同，则优先展示有效产品 / Agent 问题。
3. 如果仍相同，则优先展示高优先级问题。

每个分类下的典型 case 排序：

1. 优先展示最能代表该分类问题的 case。
2. 如果有多个候选，则优先展示高 token 消耗、用户多轮交互、明确失败、需要人工排查的 case。

## 10. 问题会话明细表字段

建议字段：

- session id
- 日期时间
- 用户名
- 邮箱
- 用户Id
- 用户场景
- 用户采集网址
- 启动次数
- 采集次数
- 采集数据量
- 导出次数
- 导出数据量
- 问题分类
- 问题原因摘要
- 是否产品 / Agent 问题
- 是否需要人工排查
- 人工排查原因
- 建议动作
- 优先级
- 备注
- 评估报告 / 评估结果链接

## 11. 每日汇总报告结构

建议结构：

1. 今日结论
2. 核心指标汇总
3. 漏斗转化分析
4. 问题会话总体情况
5. 问题分类统计
6. 各分类典型 case
7. 有效产品 / Agent 问题清单
8. 需要人工排查的会话
9. 后续建议

其中“问题分类统计”“各分类典型 case”“有效产品 / Agent 问题清单”“需要人工排查的会话分类汇总”均按问题出现频次由高到低排序。

## 12. 优先级建议

| 优先级 | 适用情况 |
| --- | --- |
| 高 | 异常 bug、Agent 流程空转、重复预览导致积分异常消耗、启动后采集失败、影响大量用户的网页识别问题 |
| 中 | 目标网站权限限制但可优化引导、用户中断但中断前有异常信号、页面结构识别不稳定、能力缺口类需求 |
| 低 | 纯用户未回应、非采集需求、正常完成、明确由用户侧条件导致且无明显产品问题的会话 |
